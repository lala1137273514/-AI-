## 💡 核心洞察

1. **评测体系五天走完一条「冷启动闭环」。** 4/13 规范 → 4/14 caseback+judge → 4/15 场景+模板+采集量 → 4/16 Langfuse RouterPlanner 闭环方案 → 4/17 字段口径+实施三件套。这不是五个孤立产出,而是一条「定义测什么 → 怎么判 → 测哪些场景 → 怎么自动跑 → 怎么落地实施」的完整链条——入职第一周就把评测从概念变成可工程化的流水线雏形。
2. **「Router 弱包含 / Planner 强匹配」断言法则是本周最具复利的方法论资产。** 它明确了不同字段应用不同严格度去判定,一个月后(5/18)正是凭这条法则识别出 bad-case「精确匹配产生假阴性」是评测方法问题而非模型 bug。早期定对判定原则,后期省下大量误判成本。
3. **「先验证能力边界、再写需求」的工程习惯在 Slack 探索上成型。** 用四类 case 把 MCP 能/不能做摸清,再谈 PRD,避免凭想象画饼——这是 W2 Slack 需求能快速落地的前置投资。
4. **Wiki 基建把「认知」沉淀成「可复用知识」。** Agent 任务与能力的飞书文档 Wiki 化(五类),让 O1-KR1 的认知对齐从「我懂了」升级为「团队能查」,是认知对齐在 DDL 节点收口的关键动作。
