1. **本周是「方法论 → 真功能」的落地执行周,旅行规划第一次把「规范→评测器→真验收」闭环完整跑穿。** W6 产出了 Loona-Spec/评测器/209 case 三段标准,本周用一条真实功能(旅行规划)验证它能落地:T1 规格(可机检)→ 轮播通道打通(可跑)→ 12 条 A1-A12 验收场景(可回归)。Agent 产品化从「方法论存在」走到「方法论被一条功能用上了」。
2. **旅行规划 T1→v2 演进体现「收敛优先于做全」的产品判断。** T1 砍掉住宿/订票/比价/地图,只复用 TravelView 两卡 + 2 工具,把一切推到 v2;v2 用 `span=ceil(总天数/5)` 统一公式替代经验分档,并把口播定位为主要内容载体(因为「轮播卡信息量小是结构性限制」)。先交「可实现+可验收+可回归的最小规格」,是 deadline 下最高效的工程化交付路径。
3. **澄清卡「暂不开发」是本周最务实的范围决策。** 距 5/30 仅剩 3 天,砍掉未定型模块、把澄清压力转移到 TTS 播报兜底(「成都三天记下了,日期/出发地/同行人/预算补不补都行」)——deadline 驱动下的优先级纪律,胜过把每个模块都做一半。
4. **loona-live 是本周隐藏的最大工程量,也是「飞书记会议、本地记工程」割裂最极端的一周。** 2 天 25 commits 从 walking skeleton 干到带 GPU 实时 STT / 混检 RAG / Docker 一键栈 / CER eval 的 OSS 级实时语音 showcase——飞书一字未记。没有本地三源(75 commits/6 仓),周报会严重低估真实产出。
5. **测试集 76.8%→85% 的瓶颈是标注准确性,不是工具能力,且强依赖刘旭单点。** 结构性噪声有两类:agent 回复不稳定致断言假阴性(~20+ 条,拟 IMM 人工介入)、响应体字段不稳定(query/limit/UID 时有时无)。把「通过率」从一个数字拆成可分层归因的工程问题,才是它能上 85% 的前提。
6. **人格化是双刃剑,本周两条人设线给出同一个工程答案:用两步拆分隔离风险。** CC 风格化把「事实保真 > 风格」立为铁律 + protected spans 1:1 保留 + 提交前两遍回读;persona 诊断出 fewshot 过拟合/过高冷要减负面约束。风格化重写最大的风险是改坏事实/数字,「先去模板再上色」把「去 AI 腔」和「注入人格」两件事分开做,是可控的解法。
